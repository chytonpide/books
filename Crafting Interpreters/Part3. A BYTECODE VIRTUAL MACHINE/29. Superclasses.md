# Superclasses 요약

메서드 상속 기능을 구현한다. 먼저 상속 구문의 컴파일을 구현한다. 
상속 구문을 발견하면 런타임에 슈퍼 클래스 스택에서 찾아서 쓸 수 있도록 수퍼클래스를 리졸빙한 정보를 얻을 수 있는 바이트코드 OP_GET_GLOBAL 과 OP_INHERIT 을 에밋한다.
런타임에 OP_INHERIT 을 인터프리팅 할 때, 수퍼클래스의 메서드 테이블을 서브클래스에 복사한다.
메서드 선언은 이 이후에 인터프리팅 되기 때문에 메서드 오버라이딩이 자연스럽게 이루어진다. 이를 copy-down inheritance 라고 부른다.
JLox 에서는 서브클래스가 슈퍼클래스에 대한 참조를 가지고 있고, 참조를 거슬러 올라가서 메서드를 찾았지만, CLox 에서는 메서드 테이블 자체가 복사됨으로 훨씬 간단하고 빠르다. 

수퍼 표현식을 사용하기 위해서 먼저 수퍼 클래스를 저장하는 기능을 구현한다. 
수퍼 표현식을 통한 수퍼 클래스에 대한 참조는 정적으로 렉시컬 스코프를 가진다. 따라서 this 와 비슷하게 상속기능을 사용하는 클래스 선언을 컴파일 할때 스코프를 열어서 super 라는 지역 변수를 만들어서,
기존 upvalue 시스템을 그대로 사용한다.

super 표현식을 컴파일 할때, super 는 단독으로 사용될 수 없고 항상 메서드 엑세스가 같이 이루어진다.
런타임에 this 와 super 가 모두 필요함으로, 이들 변수의 스택 위치를 알려주는 바이트 코드를 에밋하고, OP_GET_SUPER 와 메서드 이름을 에밋한다.
OP_GET_SUPER 를 인터프리팅 할때 스택에서 superclass 를 찾아서 메서드를 바인드하고 ObjBoundMethod 가 스택에 올라간다.

기존 메소드 호출과 마찬가지로 최적화를 진행한다. OP_GET_SUPER 와 OP_CALL 을 합친 옵코드인 OP_SUPER_INVOKE 를 정의한다.
수퍼를 통한 메서드 참조시 그대로 호출하는 코드라면 컴파일 타임에 OP_SUPER_INVOKE 를 에밋한다.
OP_SUPER_INVOKE 를 인터프리팅 할 때 마지막에 메서드 호출이 이루어져야 함으로 스택에는 메서드의 매개변수가 먼저 올라가도록 한다.
메서드 실행에는 기존 invokeFromClass 를 활용한다.

## 29.2 Storing Superclasses
- subclass 에서 superclass 의 메서드를 호출하려면 superclass 에 대한 참조를 가지고 있어야 한다. superclass 에 대한 참조는 정적이다. super 가 어떤 클래스를 가르키는지는 컴파일타임에 정해진다. 
## 29.2.1 A superclass local variable
- 스코프를 열고 super 라는 로컬 변수에 superclass 를 저장한다.
- this 는 컴파일이 초기화 될때 같이 초기화 되지만, super 지역 변수는 컴파일 도중에 초기화 되어야한다. super 라는 스트링을 토큰으로 가지고 있을 필요가 있음으로 이를 위해서 syntheticToken 함수를 정의해서 사용한다.  
- super 는 메서드 바디의 외부의 지역변수 이지만 업벨류 시스템을 통해서 VM은 메서드 내부에서 super 변수를 캡쳐할 수 있다.

## 29.3 Super Calls
- super 키워드를 컴파일 한다. this, super 지역변수를 읽는 코드와 OP_GET_SUPER 와 함께 메서드 이름을 에밋한다.
### 29.3.1 Executing super access
- OP_GET_SUPER 를 인터프리팅 한다. 스택에 superclass 가 올라가 있음으로 superclass 에서 method 를 찾고, ObjBoundMethod 를 만들어서 스택에 올린다.
### 29.3.2 Faster super calls
- OP_GET_SUPER 와 OP_CALL 을 합친, OP_SUPER_INVOKE 를 정의해서 슈퍼메서드콜을 최적화 한다. 
  - 최종적으로 실행되는 것이 메서드 발동(invocation) 이 됨으로, 스택에는 메서드의 인수가 스택에 먼저 올라가 있고, 그 위로 superclass 올라가는 형태가 된다.

## Challenges
- 1. OOP의 교리는 새 객체가 올바른 상태로 있는것을 보장하는 것 이다. 초기자 호출은 초기자를 호출 할 때, super 의 초기자 호출을 강제하도록 할 수 있다.
  필드의 경우는 만약, 서브클래스가 수퍼클래스의 필드를 덮어씌우는 경우 문제가 될 수 있다. 이 문제를 어떻게 해결할 것인가?
  - 같은 이름의 필드를 사용하는 것을 금지한다. 상속 기능을 인터프리팅 할때 super 의 this 도 만든다. 필드 set 을 인터프리팅 할때 super 의 this 를 확인한다.
  - solution: 필드에 언더바 프리픽스를 사용하서 컴파일 타임에 필드를 인지할 수 있도록 한다. 이름이 같아도 수퍼클래스의 필드와 서브클래스의 필드는 다르고 서브클래스에서 수퍼클래스의 필드에 접근할 수 없도록 한다.     
- 2. "copy-down inheritance" 상속 방법은 클래스의 선언 이후에 메소드를 수정하는것을 허용하지 않을때만 가능하다. Ruby 는 클래스 선언후에 메서드의 수정이 가능하다.
  이러한 언어의 구현에서 어떻게 클래스 변경에서도 메서드 레졸루션을 효과적으로 가능하게 하는가?
  - 슈퍼클래스에 대한 포인터 + 캐싱
    - 사본을 가지고 있지 않고, 포인터를 통해서 슈퍼클래스에 접근한다. 
    - 이렇게 하면 속도가 느려지기 때문에 캐싱을 사용한다. 이 캐싱에는 버전 정보도 저장된다.
    - 클래스를 변경하는 경우, 즉 메서드를 다시 정의하는 경우에는 메서드 테이블에서의 버전을 증가시키기 않고 버전이 일치하지 않으면 캐시를 갱신한다. 
- 3. 메서드 overriding 에서 부모에서 부터 찾기 시작하도록 기능을 구현하라, 서브 클래스의 메서드를 호출할 때는 `inner` 를 사용해서 엑세스가 가능하도록 하라.
    - `inner`호출에 ID를 부여해서 상속계층을 따라서 리졸빙이 가능하도록 한다. 
```
class A {
  foo() {
    inner();
  }
}

// The compiler desugars it to something like:

class A {
  foo() {
    this.foo@0();
  }
}    
```

※ variable(): IDENTIFIER를 읽는 파싱용 래퍼, 토큰을 읽어서 namedVariable 로 넘김
※ namedVariable(): 이 이름이 어디(로컬/글로벌/업밸류)에 바인딩되는지 결정하고 GET/SET 바이트코드를 생성
※ 이 이름이 어디에 바인딩 되는지 결정한다: 소스코드에 나온 식별자가 실행중에 어떤 저장소 (slot) 을 가르키는지 컴파일 타임에 정한다.
    - 저장소는 VM이 런타임에 값을 보관하는 실제 자리를 의미, 컴파일러는 globals, stack 의 index 를 옵코드오 함께 전달한다.
