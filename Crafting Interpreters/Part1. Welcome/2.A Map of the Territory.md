# A Map of the Territory 요약

인터프리터를 구현하는ㄴ데 있어서 책에서 다루는 여러 개념들을 먼저 살펴본다. 스캐닝, 파싱, 정적분석,
중간표현과 최적화, 코드 생성을 통해서 사용자의 코드는 비로서 실행가능한 상태가 된다.
하지만 항상 이 순서와 절차를 거쳐야 하는것은 아니고 몇몇 지점만을 통과하는 지름길도 존재한다.
AST 까지 파싱한뒤 실행하거나(JLox), 파싱 정적분석 코드생성을 번갈아 가면서 실행하는 싱글패스 컴파일러 형식도 있다. (CLox)
소스코드를 보통 더 저수준의 다른 형태로 변환하는 것을 컴파일링이라고 하고, 컴파일러는 이를 수행한 결과물을 돌려주는 것 이다.
인터프리터는 사용자의 소스코드를 바로 실행하는 것 이다. 
컴파일러와 인터프리터는 교집합을 가지고 있으면 많은 인터프리터들이 컴파일러를 내장하고 있다.

## 2.1 The Parts of a Language
### 2.1.1 Scanning
- scanning: 문자의 스트림을 덩어리로 합쳐서 토큰을 만드는 것. 어휘분석, 렉싱이라고도 부른다.
- token: 프로그래밍언어에서 의미를 가지는 "단어"(의미를 가지는 최소 단위) 같은것.
- scanner 는 스캐닝을 통해서 공백 문자같은것을 제거하고 의미있는 토큰들의 열을 만든다. 
### 2.1.2 Parsing(구문 분석)
- parsing: 토큰들의 열을 문법의 중첩된 특성을 반영하는 트리 구조로 만드는 것. 
- parser 는 파싱을 통해서 syntax trees 또는 AST 라고 부를 수 있는 것을 만든다.
  - 문법에 맞지 않는경우 parser 는 syntax error 를 돌려준다.
- 파싱의 결과 코드의 구문 구조를 알게 된다.
※ Syntax(구문)은 단어 배열의 전체 규칙을 의미한다.
### 2.1.3 Static analysis(정적 분석)
- resolution(binding): 각각의 식별자에 대해서 정의된 위치를 찾아내고 연결한다. 여기에 scope 가 작용한다.  
- scope(범위): 특정 이름이 특정 선언을 참조 하는데 사용될 수 있는 소스코드 영역.
- 정적 분석을 통해서 타입체크, 리졸빙을 수행한다. 분석결과 얻은 정보는 어딘가에 저장해야 하는데 다음과 같은 곳에 정의할 수 있다.
  - 구문트리의 어트리뷰트로 저장
  - symbol table 에 저장
  - 코드의 시멘틱을 표현하는 완전히 다른 자료구조에 저장
- 스캐닝, 파싱, 정적분석의 과정을 프론트엔드라고 부른다. 그 다음 과정은 미들엔드, 벡엔드이다.
### 2.1.4 Intermediate representations
- 컴파일러는 다음 스테이지의 구현이 더 단순해지도록, 각 스테이지에서 유저의 코드를 표현하는 데이터를 체계화 하는 파이프라인으로 볼 수 있다.
  파이프라인의 프론트엔드는 프로그램의 쓰여진 소스 언어에 특정된다. 백엔드는 프로그램이 돌게된 최종 아키텍쳐와 관계가 있다.
- 중간 단계에서는 코드는 intermediate representation 로 저장될 수 있다. 이는 source 와 destination 둘 모두와 단단하계 묶여있지않다.
  대신 IR 은 두 언어의 인터페이스로 행동한다.
- IR 은 사용장의 프로그램이 무엇을 의미하는지 이해한 상태이다. 
### 2.1.5 Optimization
- 일단 의미를 파악하면 이 단계에서 최적화 할 수 있다. 많은 성공적인 언어들은 컴파일 단계 최적하는 거의 없고 런타임 최적화에 치중되어 있다.
### 2.1.6 Code generation
- Code generation: 기계가 실제로 실행할 수 있는 형태로 만드는 단계. 여기서 code 는 cpu 가 실행하는 인스트럭션을 의미한다.
- 여기서 OS 가 cpu 로 바로 로드 할수 있는 진짜 머신 코드를 생성할 수도 있고, VM 의 코드를 생성할 수도 있다. 
- Bytecode: VM의 instructions, 언어의 의미와 더 가깝게 설계되었지만, 빽백한 이진 인코딩이다. 
  - source code: a + b 
  - real instructions : register, memory, jump
  - bytecode: LOAD_VARIABLE a, LOAD_VARIABLE b, ADD
### 2.1.7 Virtual machine
- 바이트 코드가 있다고 해도 바이트코드를 다시 네이티브 코드로 변환 하는 작업이 필요하다. 첫번째 방법은 컴파일러를 만들어서 특정 머신의 네이티브 코드로 변환하는 것이고,
  다른 하나는 가상의 아키텍쳐를 지원하는 가상의 칩을 흉내내는 Virtual machine 을 구현하는 것이다. 당연히 이는 느리지만, 대신 단순성과 이식성을 얻을 수 있다.
  예를 들어 C 로 VM 을 만들면 C 컴파일러가 있는 모든 플랫폼에서 해당 언어를 실행할 수 있다.
### 2.1.8 Runtime
- 사용자의 프로그램을 실행할때 우리가 컴파일로 머신코드를 만들었다면, OS 에게 그것을 로드하게 하면되고, 바이트코드로 만들었다면 VM 을 실행하고 그것을 로드하게 한다.
- 런타임: 완전히 저수준의 언어를 제외하고 프로그램의 실행중에는 언어가 제공하는 몇가지 서비스가 필요한데 이런 것들을 런타임이라고 부른다. 
- 완전히 컴파일 되는 언어에서, 런타임을 구현하는 코드는 결과가 되는 실행파일에 삽입된다. GO 에서는 컴파일된 응용프로그램은 Go 런타임의 복사본이 내장된다.
  언어가 VM이나 interpreter 안에서 동작한다면 런타임은 이 안에 있다.

※ : 추상 하드웨어 정의 VM, 언어가 제공하는 서비스 모음 Runtime

## 2.2 Shortcuts and Alternate Routes
### 2.2.1 Single-pass compilers
- Single-pass compiler: 파싱 및 정적 분석 코드생성을 번갈아 수행해서 구문트리나 중간표현을 할당하지 않고 직접 출력 코드를 생성하는 컴파일러.
  어떤 표현식을 만나자마자 해당 표현식을 정확하게 컴파일 하는데 필요한 모든 정보를 알고 있어야 한다.
  - C 같은 언어에서 코드를 함수를 위쪽에 선언 해야만 사용할 수 있는 작성하는 순서가 있는 이유. (예전에는 메모리가 귀했다.)
### 2.2.2 Tree-walk interpreters
- Tree-walk interpreter: AST 로 파싱한뒤 그것을 실행하는 인터프리터 → JLox
### 2.2.3 Transpilers
- transpiler(source to source compiler): 컴파일 결과를 특정 머신 코드로 만들지 않고 다른 고차원언어로 만드는 컴파일러. 이렇게 함으로써 그 언어의 컴파일러를 사용할 수 있다.
  즉 다른 소스코드를 IR 처럼 활용한다.
- Web browsers are the “machines” of today, and their “machine code” is JavaScript, so these days it seems almost every language out there has a compiler that targets JS since that’s the main way to get your code running in a browser.
### 2.2.4 Just-in-time compilation
- JIT: AOT(Ahead-Of-Time) 컴파일의 대척점에 있는 컴파일 방법으로, 런타임중에 자주실행 되는 부분을 감지하면(핫스팟) 그 부분을 기계어로 컴파일 하는 방법. 런타임에 최적화된다. 핫스팟이 아닌 부분은 인터프리팅 방식으로 실행된다.


## 2.3 Compilers and Interpreters
- What’s the difference between a compiler and an interpreter?
  - Compiling: 소스 언어를 다른것(보통 더 낮은 레벨로)으로 변환하는 구현 테크닉, bytecode, 머신코드 생성, 다른 언어로의 transpile 이 이에 해당된다.
  - Compiler: 소스코드를 다른 형태로 변환하지만 직접 실행하지 않는 것. 
  - Interpreter: 소스코드를 받아서 바로 실행하는 것. 소스코드로부터 프로그램을 구동한다.
- 컴파일러와 인터프리터의 경계는 불분명하고 교집합이 존재한다.
  - GCC: 명백한 컴파일러, 실행경과 머신코드가 생성된다.
  - Ruby 의 Matz's 인터프리터: 유저 소스코드를 AST로 만들고 실행한다.
  - JAVA 의경우
    - 사용자 시점에서는 JRE 는 인터프리터다. (중간과정이 블랙박스)
    - 내부를 뜯어보면 바이트코드변환(컴파일링), 실행에서 바이트코드를 인터프리팅 하거나, JIT으로 컴파일한 결과를 실행하기도 한다.
  - 대부분의 스크립트 언어는 인터프리터이며 컴파일러도 갖추고 있다. → Clox

## Challenges
- 3. C로 컴파일되는 Lisp 구현체는, Lisp 코드를 실시간으로 실행할 수 있는 인터프리터도 포함하고 있다. 왜일까?
  - 해답: Lisp 의 메크로는 컴파일타임에 실행해서 코드를 변환한다. 인터프리터가 없다면 C 로 컴파일 해서 실행후 확장된 소스코드를 다시 컴파일 해야한다. 이는 비용이 너무 크다.
    만약 Lisp 를 바로 실행할수 있는 인터프리터가 내장되어있다면, 메크로를 직점 실행해서 결과를 얻고 이것들을 취합해서 C 로의 컴파일(traspile)만 하면 된다.
