# Welcome
Lox 라는 고수준 스크립트 언어의 스펙을 정의하고 구현하는 것으로 프로그래밍 언어 전반에 대한 메타지식과 내가 작성한 코드가 어떻게 동작하는지에 대한 통찰을 얻을 수 있다.
표현식으로 분류되는 코드는 평가를 통해 값을 생성하는 것, 그것과 구분되는 문장은 사이드 이펙트를 생성하는 것이다.
컴파일러와 인터프리터의 차이에 대해서 배운다. 컴파일러는 소스코드를 다른 형태로 변환하는것, 인터프리터는 소스코드로부터 바로 프로그램을 실행하는 것 이다.
인터프리터가 컴파일러를 포함하는 경우가 있다. 컴파일만 할 수도 있고, 컴파일한 결과를 바로 실행할 수 있는 golang 같은 언어가 그 예이다.

# A Tree-Walk Interpreter
Java 버전인 JLox 를 구현 하면서 문법을 정의하고, 그 문법의 따라 유저의 소스코드의 표현을 변경하고(파싱) 이를 실행(인터프리팅)하는 구조를 반복적으로 구현한다.
Lox 의 문법은 재귀적인 구조를 가진다. 유저의 소스코드의 의미를 이해하려면 문법구조가 반영된 데이터 표현을 얻어야 한다. 
유저의 소스코느는 스캐닝 과정을 통해서 토큰이라는 조금더 의미있는 단위의 시퀸스가 되고, 다시 파싱을 통해서 문법의 재귀적인 특성을 반영한 AST(Abstract Syntax Tree) 라는 자료구조로 변경된다.
재귀 하향 파서(Recursive Descent Parser) 라는 라는 구현 테크닉을 사용해 파서를 구현한다. 
JLox 는 동적 타입 언어이다. 소스코드의 값의 표현은 Java 의 값 표현을 빌린다. 예를들어 Lox 의 숫자는 파싱결과 Java 의 Double 로, 문자열은 파싱결과 Java 의 String 저장된다.   
이렇게 파싱 결과 AST 형태가 된 문장의 리스트를 순차적으로 실행하는 것으로 유저의 소스코드가 '실행' 된다.

JLox 는 environment(JAVA 의 맵 자료구조) 의 체인에 전역변수와 지역변수를 저장한다.
각각의 environment 는 하나의 스코프와 매칭되고, enclosing 참조로 체인을 형성하는 구조 자체가 스코프 중첩을 가능하게 한다.
제어흐름 기능을 구현한다. 제어흐름은 선택적으로 로직을 실행시키지 않거나, 로직을 반복해서 실행 시킬 수 있는 능력이다.
AST를 재귀적으로 순회하며 실행할때, 제어 문장을 만나면 조건을 평가하고 그 결과에 따라, 다른 서로다른 문장 노드를 실행시킨다.

함수 기능을 구현한다. 함수는 코드를 추상화하고, 재사용하며, 합성 할 수 있도록 해준다.
LoxCallable 인터페이스를 정의한다. LoxCallable 인터페이스는 '호출 가능한 것'을 추상화한다. 
AST 의 함수선언 문장 Stmt.Function 을 인터프리팅 할때, LoxFunction 을 생성해서 환경에 등록한다. 
함수 실행 문장을 인터프리팅할 때, LoxFunction 은 LoxCallable 의 구현체임으로 LoxCallable 의 API를 통해서 함수 호출을 실행한다.
리턴 문장의 실행은 자바의 예외 메카니즘을 활용한다.

언어에 함수 기능을 추가하면서 새로운 문제가 발생한다. 기존의 범위(Scope) 시스템으로는 클로저(Closure) 기능을 지원할 수 없다.
클로저는 closes over 를 줄인 말로, 말 그대로 함수가 선언된 위치에서 주변 변수들을 유지하는 자료구조다.
클로저가 도입되었음으로 실행 단계에서 변수가 어떤 선언을 참조하는지 올바르게 알아내기 위해서는 파싱과 실행단계 사이의 또 하나의 처리, 즉 리졸빙이 필요하다. 
이렇듯 파싱이 완료된 상태에서 추가적인 처리를 하는 것을 정적 분석이라고 한다.

OOP 의 주요 목표는 데이터와 그에 관한 행위를 묶는 것이다. Lox 에서는 이를 클래스로 달성한다.
클래스 선언은 런타임에 LoxClass 를 생성한다.
LoxClass 은 LoxCallable 의 구현체임으로, 다시 클래스 호출 문장은 런타임에 호출되고 그 결과 LoxInstance 를 생성한다. 
객체의 프러퍼티에 할당하고 엑세스 할 수 있는 기능을 구현한다.  
메서드는 LoxClass 에 저정되어 있음으로 LoxInstance 의 메서드를 실행하려면 LoxClass의 참조를 가지고 있어야 한다.
this 표현식은 변수 표현식 처럼 다룬다.
LoxInstance에서 메서드에 접근하면, 인터프리터는 해당 메서드의 LoxFunction을 그대로 돌려주는 것이 아니라 this를 현재 인스턴스에 바인딩한 새 LoxFunction을 생성하여 돌려준다.
인터프리터가 올바르게 바인딩을 할 수 있도록 리졸버에서도 this 대한 스코프를 설정해야 한다.

상속기능을 구현한다. 클래스 선언의 문장에 상속을 표현할 수 있도록 필드를 추가한다.
상속을 한다는것은 superclass 에서 참인 모든 것은 subclass 에서도 참이어야 한다는 것이다.(리스코프 치환법칙)
오브젝트의 메서드에 엑세스할 때 먼저 클래스의 메서드를 먼저 찾고 없으면 superclass로 올라가서 찾는다.
superclass 의 메서드에 대한 엑세스는 super 표현식으로 제공한다.
이때 클로저가 동작하는 것 처럼 클래스의 선언에서 super 를 캡쳐해야 한다.
super 를 통해 엑세스한 메서드 내에서도 this 는 현재 오브젝트를 가리키고 있어야 한다.


# A Bytecode Virtual Machine
Lox 의 C 로 구현된 인터프리터인 CLox를 구현한다. 목적은 JLox 보다 더 빠른 인터프리터이다. 
이를 달성하기 위해서 유저의 소스코드의 파싱결과를 AST 가 아닌 바이트코드로 표현한다. 
바이트코드는 VM(가상 머신) 에서 실행될 수 있는 이진 인스트런션의 선형 나열로, 특정 cpu 아키텍처에 종속되는 머신코드(컴파일 결과 생성되는)와 비교해서 생각 할 수 있다.
당연히 머신코드보다는 속도가 느리지만 vm 위에서 실행되기 때문에 cpu 아키텍쳐에 독립적이어서 이식성이 크다.
표현의 변경이 중대한 차이를 만든다. 바이트코드는 중첩 트리 구조가 아닌, 평탄화된 시리즈의 구조를 가진다.
데이터는 조밀해져서 캐시 친화적이고 성능은 향상되지만 이 자료구조를 사용하기 위한 구현은 더 복잡해진다.

또다른 주요한 차이점은 JLox 는 스캐닝과 파싱 실행이 각각의 단계(pass)로 나뉜다. 각 단계마다 중간 표현이 생성되고 이것을 메모리에 저장해 두었다가 다음 단계에서 순회한다.
C에서는 이와 같이 각 단계의 결과물을 별도로 저장하기 위해서는 메모리 관리를 위한 많은 추가적인 구현이 필요하다.
따라서 CLox 에서는 유저의 소소코드를 단 한번만 훑으면서 중간 표현을 저장하지 않고 바로 바이트코드를 생성하는 싱글 패스 컴파일러 라는 구현 태크닉을 사용한다.
Lox 는 적은 맥락으로도 구문 문법을 이해할 수 있게 설계된 언어이기 때문에 이것이 가능하다.
여기서 오해하지 말야하 할 것은 스캐닝과 파싱이 싱글 패스로 이루어진다는 것이다. JLox 에서 문장단위로 AST를 실행하듯, CLox 에서도 문장단위로 바이트코드를 실행하며 순회한다.

이제 CLox 에서는 유저의 소스코드를 바이트코드로 변경함으로 이 과정을 컴파일로 부를 수 있다.
CLox는 프랫 파서(Pratt Parser)라는 구현 테크닉을 사용해서 표현식을 파싱한다. JLox의 재귀하향 파서가 우선순위를 함수 호출 체인의 깊이로 인코딩하는 것과 달리, 
프랫 파서는 명시적인 우선순위 테이블을 두고 하나의 parsePrecedence() 함수에서 재귀적으로 파싱을 수행한다.
싱글 패스 컴파일러이기 때문에 파싱 결과는 문법의 재귀적 특성 자체를 현시한 것과 같은 AST가 아닌 플랫한 바이트코드 시퀀스다.

바이트코드는 옵코드(opcode)와 오퍼랜드(operand)의 시퀀스다. 옵코드는 VM이 수행해야 할 연산의 종류를 나타내고, 오퍼랜드는 그 연산에 필요한 추가 데이터다.
VM은 바이트코드를 실행할 때 디스패치 루프를 통해 하나씩 옵코드를 읽고(fetch), 해당 옵코드에 대응하는 처리로 분기(dispatch)하여 실행하는 것을 반복한다.
VM이 이 바이트코드를 실행할 때, 중간 계산값의 생성·소비 순서가 LIFO와 일치하기 때문에 스택을 사용하여 관리한다.

CLox는 Lox의 동적 타입 값을 표현하기 위해 tagged union 기반의 Value 구조체를 정의한다. Tagged union은 하나의 메모리 공간(union)에 여러 타입의 값 중 하나를 담고, 태그 필드로 현재 어떤 타입인지 구분하는 패턴이다.
JLox 에서 파싱 결과 소스코드의 값이 Java 의 값 표현으로 저정된 것 처럼, CLox 에서 컴파일의 결과 Value 타입으로 저장된다.
문자열처럼 크기가 가변적인 데이터는 union에 직접 담을 수 없으므로, 힙에 할당하고 포인터만 Value에 저장한다. 힙에 할당되는 모든 객체의 공통 헤더로 Obj 구조체를 정의하고, ObjString 등이 이를 첫 필드로 포함하여 일종의 상속 구조를 구현한다.

변수 기능을 구현하기에 앞서서 변수이름으로 값을 조회할 수 있는 기능을 구현해야한다. JLox 에서는 Java 의 HashMap 을 공짜로 사용했지만 C 에는 그런것이 없음으로 해시 테이블(HashTable)을 직접 구현한다.
해시 테이블은 키와 값의 페어로 이루어진 엔트리의 배열로 키를 통해서 값을 찾을 수 있는 자료구조이다. 이 자료구조의 핵심적인 기능은 엔트리 수가 아무리 많아도 상수시간 안에 값을 찾을 수 있다는 것이다.
배열의 크기가 정해져 있을때, 임의의 길이의 키를 제약없이 사용하기 위해서는, 키를 배열 인덱스 범위로 매핑해야 한다. 이때 사용되는 것이 해시 함수이다.
임의의 두 키의 해시값이 같은 충돌(collision)이 일어날 수 있음으로 이를 해결하기 위한 메카니즘이 필요하다. 오픈 어드레싱 방식중, linear probing 이라는 메카니즘을 사용해 이를 해결한다.

전역변수는 선언시, 런타임에 VM에 의해서 전역변수 테이블에 키와 값이 쓰여진다. 선언의 우변식은 이미 평가되어 스택에 올라가 있는 것이 값, 청크의 상수테이블에 저장되어 있는 문자열이 키 이다.
전역변수를 사용할 때(엑세스 하거나, 할당할 때)는 단순히 전역변수 테이블에서 값을 찾아서 읽은뒤 스택에 값을 올리거나, 마찬가지로 할당의 우변식이 평가된 값으로 테이블의 값을 덮어 씌운다.
여기서 선언은 문장임으로 스택에 값을 남기지 않고, 엑세스는 표현식으로 스택에 값을 남기고, 마찬가지로 할당도 Lox 에서는 표현식 이므로 스택에 값이 남는다.
전역 변수의 경우 컴파일이 끝난 후에 런타임에 VM 에서 리졸빙 된다. 단순히 vm이 동작하는 런타임에 키로 값을 읽거나 쓰거나 하는 식으로 비교적 단순하고 명확하게 동작한다.

지역변수는 훨씬 전역변수보다 더 많이 사용됨으로 이보다 더 효율적으로 동작하도록 구현한다.
지역변수의 스코프, 즉 생성과 소멸은 스택의 LIFO와 일치한다는 이 사실을 활용한다. 
지역변수는 컴파일 타임에 리졸빙 된다. 지역변수의 선언에서 변수의 이름과 스코프의 깊이는 컴파일러에 저장된다.
놀랍게도 런타임에서는 지역변수 선언을 위한 별도의 동작은 없다. 선언을 포함한 문장이 vm 에서 실행되면, 선언의 우변식은 평가되어 스택에 올라간다. (초기값이 없는 경우에도 NIL_VAL 값이 스택에 올라가 있다.)
지역변수를 사용할때 컴파일러는 자신이 가지고 있는 지역변수의 인덱스 값을 옵코드의 피연산자로 전달한다.
런타임에 VM은 이 인덱스를 활용해서 스택에서 값을 가지고 오는데, 이는 지역변수의 값이다. 컴파일러에서 추적하는 지역변수 배열의 인덱스와 스택의 슬롯이 일치한다는 사실을 이용하는 것이다.
이렇듯 전역변수가 런타임에 이름을 명시적으로 사용한것과 달리, 선언한 지역변수의 값이 스택에 남아있는 남아있는 사실을 활용해서 암시적으로 인덱스를 통해 지역변수를 특정한다.
컴파일타임과 런타임에 걸친 동작은 이해하기는 힘들지만, 배열의 인덱스를 통해서 값에 접근함으로 결과적으로 더 빨라진다.

제어 흐름에서 흐름은 실행이 프로그램의 텍스트를 이동하는 방식을 의미한다. JLox 에서는 실행할 코드조각이 AST 의 노드로 존재하고 런타임에 암시적으로 결정 되었다.
CLox 에서 vm 은 현재 실행중인 바이트코드 인스트럭션의 주소를 ip 를 통해 관리한다. ip(Instruction Pointer, Program Counter) 는 정확하게 프로그램이 지금 어디에 있는지를 표현하고,
이를 조작하는 것으로 제어흐름을 구현할 수 있다.
제어흐름문장을 컴파일 할때 컴파일러는 ip 를 조작하는 점프 계열의 OpCode 를 청크에 쓴다. 당연히 어디로 점프할지는 피연산자로 주어져야 한다.
CLox 는 싱글패스 컴파일러다. 만약 if 조건문을 컴파일 한다고 하면, 조건을 만족 할 때 실행할 문장을 컴파일 하기 전까지는 어디로 점프할지 피연산자를 알 수 없다.
따라서 점프 계열의 OpCode 와 피연산자의 플레이스홀더를 청크에 쓴 후, 문장을 컴파일 해서 점프할 위치를 알아낸 다음, 이 값을 피연산자로 덮어쓰기한다. 이러한 테크닉을 백패칭이라고 한다.

VM에 입장에서 함수는 실행가능한 바이트코드의 덩어리이다. 그리고 Lox 에서 함수는 일급 객체(변수에 할당 가능함으로) 임으로 Chunk 필드를 가지고 있는 ObjFunction 로 표현한다.
함수 기능을 도입하는 것으로 컴파일 실행의 구조가 변경된다. 최상위 수준에서 유저의 소스 코드의 컴파일 결과는 함수가 되고 VM이 그것을 실행하는 구조가 된다. 
소스코드 내부에서의 함수의 선언을 만날때 마다 새로운 컴파일러가 초기화 되고 이 컴파일러들은 함수의 지역변수를 관리하게 된다. 컴파일러들은 링크드 리스트형태로 연결된다.
각각의 함수는 각각의 컴파일러에 의해서 컴파일 되고 함수 본체는 상위 컴파일러의 청크의 상수테이블에 쓰여지고, 이름은 OP_DEFINE_GLOBAL 옵코드로 청크에 에밋된다.
(중첩된 함수는 지역변수로 컴파일 된다.)
여러 함수에 걸쳐서 로컬변수가 정의된다고 하더라도, 나중에 정의된 변수가 먼저 정의된 변수보다 폐기된다는 원리를 따르고, 따라서 여기에도 스택을 사용할 수 있다.
이때 매개변수를 포함해서 함수의 로컬 변수들이 저장되는 스택의 토막을 콜 프레임(call frame) 이라고 한다.
함수 호출을 컴파일 하면 OP_GET_GLOBAL 과 OP_CALL 옵코드가 에밋된다. VM 은 전역변수 테이블에서 함수 본체를 찾아와서 스택에 올리고 이를 참조하는 CallFrame 을 생성한다.
VM 이 최상위 함수인 run() 이 참조하는 frame 지역변수를 새로운 CallFrame 으로 교채하는것으로 VM 함수의 청크를 실행할 수 있도록 한다.
함수의 청크를 실행함에 있어, 함수의 지역변수에 접근할때 CallFrame 의 slots 포인터를 기준으로 상대적인 인덱스를 사용하는것으로 지역변수를 가르키는 스택 슬롯에 정확하게 접근할 수 있다.
함수의 청크 에는 반드시 OP_RETURN 이 마지막에 쓰여있음으로 이것을 해석할때 다시 run() 이 참조하는 frame 을 교체하는 것으로 상위 흐름으로 돌아올 수 있다.
** VM 은 지역변수와 계산 중간값은 여전히 전역적인 stack 을 통해서 관리하지만, 각 호출시의 소유권을 표현하기 위해서 추가적으로 CallFrame 의 배열을 사용한다. ** 


클로저 기능을 구현한다. JLox 에서는 로컬 변수를 환경(Environment) 에 저장하고, 환경의 체인을 유지하는 것으로 별다른 메카니즘 없이, 리졸빙 패스를 통해서 변수를 추적할 수 있었다.
CLox 에서는 로컬 변수를 스택에 저장한다. 따라서 클로즈를 위한 별도의 메카니즘을 구현한다.
먼저 클로저를 위한 인프라 스트럭쳐를 구현한다. 클로저를 ObjClosure 로 표현한다. **기존 vm 이 ObjFunction 을 실행했지만 이 사이에 ObjClosure 라는 레이어가 추가된다.**
컴파일러는 이제 함수 선언을 컴파일 할 때 OP_CLOSURE 를 에밋한다.

### 클로저가 없을 때의 파이프라인 
```
컴파일: ObjFunction 생성 → 상위 chunk의 상수 테이블에 저장
런타임: OP_CONSTANT → ObjFunction을 스택에 push
        OP_DEFINE_GLOBAL → 글로벌 테이블에 저장
호출:   OP_GET_GLOBAL → ObjFunction을 스택에 push
        OP_CALL → CallFrame 생성 (ObjFunction 참조)
```
### 클로저가 추가된 파이프라인
```
컴파일: ObjFunction 생성 → 상위 chunk의 상수 테이블에 저장 (여기까지 동일)
        + upvalue 정보를 ObjFunction에 기록
런타임: OP_CLOSURE → ObjFunction을 ObjClosure로 감싸서 스택에 push
        OP_DEFINE_GLOBAL → ObjClosure를 글로벌 테이블에 저장
호출:   OP_GET_GLOBAL → ObjClosure를 스택에 push
        OP_CALL → CallFrame 생성 (ObjClosure 참조)
```

클로저가 캡쳐한 자기 함수 바깥의 로컬 변수의 대한 참조를 upvalue 라고 한다. ObjClosure 는 이를 필드로 가진다.
이 로컬 변수가 아직 스택에 남아 있으면 이를 open upvalue, 이 로컬 변수가 포함된 스코프가 닫혀서 스택에 남아 있지 않아서 힙으로 옴기면 이를 closed upvalue 라고 부른다.
Lox의 지역변수는 렉시컬 스코프를 가진다. (소스코드 상에서 스코프가 정해진다.) 따라서 컴파일 시점에서 upvalue 를 위한 정보를 파악할 수 있다.
컴파일러는 함수 선언을 컴파일 할 때 OP_CLOSURE 와 함께, 피연산자로 isLocal과 index 를 에밋하고, 나중에 vm 에서 올바른 수 만큼 피 연산자를 읽도록 컴파일러에 upvalue 의 갯수를 기록한다.
VM은 OP_CLOSURE 를 실행할 때 이 정보를 바탕으로 ObjClosure 를 초기화 한다. ObjClosure 에는 upvalue 의 런타임 표현인 ObjUpvalue 의 이중 포인터를 필드를 가지는데, 이것의 초기화도 진행한다.
isLocal이 true면 바로 바깥 함수의 스택 슬롯을 직접 캡처하고, false면 바깥 클로저의 upvalue 배열에서 가지고온다.
upvalue 들은 중복을 방지하기 위해서 전역적으로 VM의 openUpvalues 필드로 관리된다.

컴파일러에서 변수를 리졸브 할때, 그것이 upvalue 이면 OP_GET_UPVALUE, OP_SET_UPVALUE 를 에밋한다. 
VM 에서는 이 옵코드를 실행할때 선언에서 초기화된 ObjClosure 의 upvalues 필드에서 변수의 위치를 읽어온다.

upvalue 의 클로징 기능을 구현한다. 먼저 두가지 질문에 답을 할 필요가 있다. 클로즈드 오버 된 변수를 어디에 저장할 것인가?, 언제 upvalue 를 클로즈 할 것 인가?
첫번째 질문의 답변은 ObjUpvalue 자체가 object 시스템을 사용해서 힙에 존재함으로 ObjUpvalue 에 closed 필드를 추가한 후에 여기에 값을 저장하는것이다.
두번째 질문의 답변은 스코프를 종료할 때다. 컴파일러가 스코프를 종료할때 OP_CLOSE_UPVALUE 를 에밋한다.(isCaptured 플래그를 통해서, upvalue가 있을 경우에만)
VM 에서는 OP_CLOSE_UPVALUE 를 실행할 때, VM 의 openUpvalues 에서 닫히는 스코프에 포함된 upvalue 만 close 한다. openUpvalues 에서 노드도 제거된다.
VM 에서 OP_RETURN 을 실행할 때도, upvalue 를 클로즈 해야 한다.
close 를 구현하는 방법은 단순하다. ObjUpvalue 가 값 자체를 가지도록 하고, location 이 자기 자신이 가지고 있는 값을 가리키도록 하는 것이다.

JLox 에서는 JVM이 메모리르 관리해줌으로 이에 관해 무지한 채로 있을수 있었다. 
CLox 에서 객체들은 malloc 을 통해서 실제로 힙에 할당된다. 따라서 더이상 사용하지 않는 메모리를 회수하는 컴퍼넌트가 필요하다. 이를 가비지 컬렉터라고 한다.
먼저 메모리의 어떤 부분이 사용되지 않는지 알아내야 한다. 유저 프로그램이 어떤 방식으로든 참조 할 수 있는 값을 reachable 이라고 하고,
unreachable 한 값들이 저장되어 있는 메모리를 회수해야 한다. 이를 Reachability 라고 한다.
Mark sweep Garbage Collection 알고리즘을 사용한다. 이는 VM이 직접 접근할 수 있는 root 부터 시작해서 도달 가능한것들에 순회를 하면서 마킹을 하고, 마킹이 없는 것들의 메모리를 해제 하는 알고리즘이다.
컴파일 과정에서 등장하는 객체들도 GC 에 의해서 관리되도록 VM의 stack 임시적으로 올려야한다. stack 은 root 임으로 GC가 reachable 이라고 판단할 수 있다.  
언제 가비지 컬렉션을 실행할지도 정해야한다. 적절한 타이밍은 힙 할당이 일어날 때이다.
GC 의 효율성을 측정하는 메트릭으로 throughput 과 latency 가 있다. 이는 트레이드 오프 관계이다.
- throughput: GC 작업을 하는데 사용하는 시간조각 대비 유저 코드를 동작 시키는데 사용하는 시간조각. ex) throughput 90% 면, user 프로그램에 90% 의 시간을 쓰고 10% 는 GC 간접비가 된다.
- latency: GC 작업 때문에 유저의 프로그램이 멈추는 가장 긴 연속적인 시간.
Clox 의 vm 은 throughput 과 latency 의 트레이드 오프 조정에 자동 조정(self-adjusting)을 사용한다. 직전 힙의 사용량이 다음 GC 실행의 threshold 를 결정하게 한다.

클래스 기능을 구현한다. 이를 통해 록스의 표현력을 향상 시킨다. 클래스와 인스턴스의 런타임 표현힌 ObjClass 와 ObjInstance 를 구현한다.
클래스의 선언을 컴파일 할때 컴파일러는 OP_CLASS, OP_DEFINE_GLOBAL 를 순서대로 에밋한다. 
VM 은 OP_CLASS 를 실행해서 ObjClass 를 만들고 이어서 OP_DEFINE_GLOBAL 에 의해서 vm 의 전역 해시테이블에 클래스 오브젝트가 설정된다.
VM 에서 ObjInstance 의 생성은, OP_CALL 이 실행될때 callee의 타입을 확인해서 ObjClass인 경우 ObjInstance를 힙에 생성하고, 스택에서 클래스 자리를 인스턴스로 교체하는것으로 이루어진다.
인스턴스는 상태를 가진다. ObjInstance에 해시테이블 타입의 fields 라는 필드를 추가한다. 
인스턴스에 대한 get, set 표현식을 컴파일 한다. 컴파일 결과 OP_GET_PROPERTY, OP_SET_PROPERTY 를 에밋한다.
VM 에서 이를 실행할때, 스택에 이미 설정된 ObjInstance 를 찾고, 이어서 set 인경우 해시테이블에 값을 할당하고(할당은 표현식임으로 할당한다음 스택에 값도 올린다.), get 인 경우 값을 읽어와서 스택에 올린다.

메서드 기능을 구현한다. 메서드는 클래스에 위치한다. ObjClass 에 해시테이블 타입의 methods 라는 필드를 추가한다.
클래스의 선언을 컴파일 할때 메서드를 선언하는 코드를 발견하면 이를 함수를 컴파일하는 코드로 컴파일한다. 이는 OP_CLOSURE 를 에밋한다. 메서드임으로 추가로 OP_METHOD 를 에밋한다.
OP_METHOD 를 실행할 때 OP_CLOSURE 에 의해서 스택에 올라와 있는 함수 본체를 ObjClass 메서드 필드에 설정한다.
메서드를 참조를 가능하게 한다.Lox 에서 메서드는 일급 객체로 변수에 할당하는 것이 가능하다. 메서드를 실행할 때 this 는 올바른 객체를 객체를 가리켜야 한다.
메서드를 실행할 때 this 를 통해 접근하는 대상 객체를 메서드의 리시버라고 한다. 따라서 메서드를 변수에 할당할 때, 메서드는 리시버에 대한 정보를 가지고 있어야 한다.
VM 에서 메서드 접근을 실행할 때, 예를들어 `var a = instance.aMethod` 와 같은 할당이라면 리시버에 대한 정보를 가지고 있는 런타임 표현인 ObjBoundMethod 를 만들어서 스택에 올린다.
ObjBoundMethod 는 ObjClosure 를 레핑한다. 이후 OP_SET_LOCAL 을 실행하면서 콜 프레임의 슬롯에 ObjBoundMethod 가 설정된다.
메서드를 컴파일할때, this 를 slot 0의 지역변수로 컴파일 한다. (initCompiler() 에서 slot 0의 이름을 "this"로 등록한다.)
instance.method 를 평가할 때 인스턴스를 리시버로 하는 ObjBoundMethod 를 생성한다.
ObjBoundMethod 를 호출할 때, 리시버를 slot 0 에 설정한 다음 ObjClosure 를 호출한다.
따라서 메서드 본문에서 this(OP_GET_LOCAL 0)에 접근하면 리시버 인스턴스에 접근하게 된다.
인스턴스가 생성될때부터 올바른 상태를 가지도록 초기자 기능을 구현한다. ObjClass 를 호출할때 init 이라는 메서드를 찾아서 실행하도록 한다.
init 가 발동될때 메서드는 항상 this 를 리턴한다.
없으면 기존 루틴대로 ObjInstance 만 생성해서 스택에 올린다. 
컴파일은 기존 메서드 컴파일을 그대로 사용한다. 메서드 이름이 init 인 경우, 이를 초기자로 보고 마킹한다.
메서드는 대부분의 경우 변수에 할당되지 않고 그자리에서 실행된다. OP_INVOKE 라는 전용 옵코드를 통해서 매서드 발동(실행)을 최적화 한다.
OP_INVOKE 를 통해서 바로 메서드를 실행하는 경우에도 this 가 올바른 리시버 를 참조할 수 있도록 해야한다. 인스턴스가 이미 스택에 올라가 있으므로 그것을 활용한다.
```
메서드 바로 발동 → instance 가 평가되어서 스택에 올라가 있기 때문에, 기존 스택상태 그대로 사용
메서드를 변수에 할당한다음 나중에 발동 → 할당 시점에 receiver 를 저장해 두기 위해 별도의 래퍼(ObjBoundMethod) 필요, 발동할 때 this 변수가 올바르게 리졸브 되도록, 스택의 slot 0 를 receiver 로 변경해야함.
```

메서드의 상속 기능을 구현한다. 클래스 선언에서 상속 구문을 발견하면 런타임에 슈퍼 클래스를 스택에서 찾아서 쓸 수 있도록 OP_GET_GLOBAL 과 OP_INHERIT 을 에밋한다.
VM 은 OP_INHERIT 을 실행할 때, 수퍼클래스의 메서드 테이블을 서브클래스에 복사한다.
메서드 선언은 이 이후에 인터프리팅 되기 때문에 메서드 오버라이딩이 자연스럽게 이루어진다. 이를 copy-down inheritance 라고 부른다.
JLox 에서는 서브클래스가 슈퍼클래스에 대한 참조를 가지고 있고, 참조를 거슬러 올라가서 메서드를 찾았지만, CLox 에서는 메서드 테이블 자체가 복사됨으로 훨씬 간단하고 빠르다.

슈퍼 식별자를 사용하기 위해서 먼저 슈퍼 클래스에 대한 참조를 저장하는 기능을 구현한다.
super 식별자가 어떤 클래스를 참조하는 지는 소스코드상에서 이미 정해진다. 즉 정적 렉시컬 스코프를 가지고 있다.
따라서 컴파일 타임에 super 라는 지역변수를 만들어서 메서드의 바깥 스코프의 변수 즉 upvalue 에 저정한다.

super 는 단독으로 사용될 수 없고 항상 메서드 엑세스가 같이 이루어지는 표현식이 된다.   
`super.method` 라는 표현식을 컴파일 하면, OP_GET_LOCAL(this), OP_GET_UPVALUE(super), OP_GET_SUPER 가 순서대로 에밋된다.
VM 은 OP_GET_SUPER 를 실행할 때, 스택에 올라가있는 수퍼 클래스에서 메서드를 찾고, 마찬가지로 인스턴스를 리시버로 설정하면서 ObjBoundMethod 를 만든다.
기존 메소드 호출과 마찬가지로 최적화를 진행한다. OP_GET_SUPER 와 OP_CALL 을 합친 옵코드인 OP_SUPER_INVOKE 를 정의한다.

일반적인 메서드 엑세스에 에서, `var closure = instance.method` 를 평가하면 ObjBoundMethod 의 receiver 는 인스턴스 자체가 되고, 메서드 본문에 this 식별자가 가 있는경우 이를 가르키게 된다.
슈퍼 메서드 엑세스에서 `var closure = super.method` 를 평가하면 ObjBoundMethod 의 receiver 는 이 표현식을 감싸고 있는 메서드의 인스턴스가 되고, super 메서드의 본문에 this 식별자가 있는 경우 이를 가르키게 된다.  

최적화는 프로그램의 성능을 개선하는 것이다. 최적화를 하면 프로그램은 더 적은 리소스(실행시간, 시작시간, 메모리 사용량등)를 사용하게 된다.
최적화가 성능을 얼마나 개선했는지, 최적화에 의한 변경에 의해서 다른 영역의 성능을 저하시키지 않았는지 검증할 수 있는 프로그램을 밴치마크 라고 부른다.
프로그램을 실행하고 코드가 실행되는동안 하드웨어 리소스 사용량을 추적하는 도구를 프로파일러라고 부른다.

CLox 에서 최적화를 진행한다. 헤시테이블의 프로빙을 빠르게 만든다. 해시값을 계산할 때, hash % capacity 의 모듈로 연산을 한다.
제수가 2의 거듭제곱일 경우, 피제수와 제수를 2진수로 표현했을 때, 피제수에서 제수의 최상위비트 이상은 나누어떨어지는 수이고, 나머지 비트가 모듈로 연산의 결과가 된다.
이는 제수 -1 과 피제수의 AND 연산과 같다. 그리고 capacity 는 8로 시작해서 2배씩 증가하는 2의 거듭제곱이다.
이 사실을 이용하면 모듈로 연산을 훨씬 빠른 AND 연산으로 대체할 수 있다.

NaN 박싱 기법을 사용해서 값의 크기를 감소시켜 캐시 친화적으로 만드는 것으로 성능을 개선한다.
NaN 박싱은 VM 에서 사용하는 가장 큰 값 타입인 더블을 기준으로 나머지 다른 타입의 값들을 NaN 의 비트 패턴(quiet NaN 사용되지 않는 여분의 비트) 으로 인코딩 하는 방법이다.
